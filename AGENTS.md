# AGENTS.md

> Codex 项目级工作说明。任何代码、内容、测试或文档的创建与修改，都应先阅读并遵守本文件。

# 1. 项目身份

## 1.1 工作标题

**《取经路线非官方注释》**

英文内部代号：`unofficial-journey-map`

## 1.2 一句话愿景

本项目不是一张干净的“西游记取经路线图”，而是一张会争吵的文学地图：它的边缘不断被小妖、无名随从、村庄传闻、破损遗物、官僚删改、尸体、梦境与空白侵入。

## 1.3 主要交付物

构建一个完整的浏览器端互动文学游戏 / 数字人文作品，至少包含：

- 一张横向卷轴式取经路线地图。
- 取经重要节点的丰富档案页。
- 一个由正注、旁注、反注、误注、梦注、账簿注、尸体注、译注、地图注、留白注构成的批注系统。
- 一个会根据玩家编辑选择而反馈的审稿 / 审查 / 官僚批改系统。
- 一个数据驱动的档案库，收录人物、地点、证据、声音、资料来源与术语。
- 一个基于玩家在正本度、旁注度、余声、驳杂、审稿风险、留白感等数值之间取舍的多结局系统。
- 若干嵌入路线节点的小型、精致、有主题意义的互动模块。
- 一个内容丰富、具有文学性、可游玩、可演示、适合作为严肃课程项目展示的最终作品。

## 1.4 核心隐喻

玩家是一名临时档案员 / 外部批注员，被某个暧昧的机构雇用，例如：

- “西行路线整理局”
- “灵山取经工程后评估办公室”
- “三界地理与功德校勘署”

表面的任务是整理一张清晰、可靠、可审查的取经路线图。

真正的游戏，是决定那些无法被干净路线容纳的东西将如何存在。

# 2. 不可妥协的设计原则

## 2.1 不要做成普通路线浏览器

本产品不能只是时间线、百科式地图或章节摘要网站。每个核心功能都必须强化项目的中心冲突：

> 一张清楚的地图让取经路变得可读，却也会抹掉那些不适合进入路线的人生。

## 2.2 文学与系统必须不可分割

玩法系统不是装饰，它们必须表达文学观念：

- 把一张“尸体注”拖进边缘，不只是 UI 操作，而是一种伦理选择。
- 接受审稿意见，不只是点击按钮，而是与经典叙事的简化机制合作。
- 留白注不是缺少内容，而是拒绝替死者或沉默者说完。
- 地图不是中性的，它是一种选择、权威、记忆和删除的技术。

## 2.3 低美术成本，高系统密度，高文本质感

默认人类创作者不擅长大量手绘插画。避免依赖大规模定制角色立绘或复杂场景美术。

优先视觉语言：

- 古籍、木刻、纸纹。
- SVG 路线、朱印、印章。
- 极简剪影。
- 手写边注。
- 官僚档案 UI。
- 数据卡、文书碎片、印章、线绳、破损册页。
- 符号化图标，而非精细插画。

即使只使用简单形状、CSS、SVG、emoji 和文字，作品也应该看起来是有意设计过的，而不是临时拼贴。

## 2.4 默认数据驱动

重要内容必须放入结构化数据文件，不要硬编码在 React 组件内部。

使用数据文件管理：

- 路线节点。
- 批注卡。
- 审稿意见。
- 证据物件。
- 声音 / 发言者。
- 结局。
- 资料来源。
- 术语表。
- 小游戏元数据。

设计应用时，应允许未来新增节点和批注，而不需要改动核心 UI 逻辑。

## 2.5 从一开始就内容丰富

不要止步于空骨架。不要只创建 placeholder 文本。不要交付一个“系统存在但文学内容缺席”的玩具。

一个可展示的版本应包含：

- 一张有相当数量节点的路线地图。
- 多个完整写作过的节点档案。
- 多种类型、分布在不同节点的批注卡。
- 多个审稿事件。
- 多个结局。
- 若干能展示项目可玩语法的独特互动场景。

当时间或环境受限时，优先保证端到端可玩的完整作品、深入文本内容和稳定系统，而不是视觉炫技。

## 2.6 不要让用户填明显空白

当细节缺失时，做出合理的文学或技术假设，直接实现，并在 `docs/ASSUMPTIONS.md` 或 README 中记录该假设。

只有当某个决策会根本改变项目设定时，才向用户询问。

## 2.7 课程项目语境

这是一个课程项目，也是解释性 / 教育性原型。不要因为商业 IP 谨慎而降低概念野心。项目可以引用《西游记》、相关衍生作品和当代改编作为概念影响。仓库仍应保持自包含：避免依赖远程第三方资源、热链接图片、付费字体或不稳定外部 API。优先使用原创占位、CSS/SVG 视觉和清晰分离的资产挂钩。

# 3. Codex 工作规则

## 3.1 修改文件前

1. 检查仓库结构。
2. 完整阅读 `AGENTS.md`。
3. 如果存在，阅读 `README.md`、`package.json`、已有 docs 与核心源码。
4. 判断项目当前阶段。
5. 保留用户已有修改，不要盲目覆盖。
6. 编辑前形成具体实施计划。

## 3.2 工作循环

每个主要任务按以下循环推进：

1. **理解**：检查相关文件和数据。
2. **规划**：形成实现方案。
3. **实现**：进行连贯、增量的代码和内容修改。
4. **充实**：加入有意义内容，不只搭框架。
5. **测试**：运行 typecheck、单元测试、构建和相关 UI 检查。
6. **复核**：检查 diff，避免回归、死代码和浅薄文本。
7. **文档**：当行为变化时更新 README / docs / 数据 schema。
8. **迭代**：测试失败则修复，内容单薄则补写。

## 3.3 完成标准

任务不能因为 app 能编译就算完成。任务完成意味着：

- 功能端到端可用。
- 实现具有类型安全。
- 界面可读、可操作。
- 功能包含真实内容，并体现项目文学前提。
- 既有测试通过；非平凡逻辑有新增测试。
- 构建成功。
- README 或 docs 反映了新行为。

## 3.4 优先命令

使用 `package.json` 中定义的脚本。如果尚不存在，请创建。

推荐脚本：

```bash
npm run dev
npm run build
npm run preview
npm run typecheck
npm run lint
npm run test
npm run test:watch
```

若某个脚本缺失，要么添加它，要么说明为什么不适用。

## 3.5 错误处理

- 修复根因，不要只掩盖症状。
- 除非有记录原因，不要用 `any` 压制 TypeScript 错误。
- 不要全局关闭 lint 规则来消灭警告。
- 不要留下坏掉的 import、死路由或大型未使用组件。
- 如果使用 `localStorage` 等浏览器 API，需要为 SSR / 测试环境做保护。

## 3.6 每次主要实现后的自检清单

检查：

- 这个功能是否强化了“官方路线 vs 非官方批注”的冲突？
- 是否加入了足够真实的文本内容？
- 用户不读源码也能理解要做什么吗？
- 全局数值是否可见且可解释？
- 批注是否能跨刷新保存？
- 不同玩法风格是否能抵达不同结局？
- 构建是否通过？
- 是否避免把项目做成普通 wiki？

# 4. 推荐技术架构

## 4.1 核心技术栈

使用：

- Vite
- React
- TypeScript
- SVG 作为主文学地图
- CSS Modules 或普通 CSS + CSS variables
- JSON 或 TypeScript 数据模块承载内容
- React Context + `useReducer` 管理全局游戏状态，除非有更强理由引入状态库
- `localStorage` 保存玩家进度
- Vitest 做逻辑测试
- React Testing Library 做组件测试

在依赖确实提升项目之前，避免增加重型库。

## 4.2 可选库

仅在有明确收益时使用：

- Phaser 或 Canvas：用于火焰山风向模拟等嵌入式小游戏。
- D3：用于关系图，不用于主路线地图。
- MapLibre：仅在实现历史玄奘路线 overlay 时使用。不要让真实 GIS 成为核心界面。
- Framer Motion：仅在动画确实重要且可维护时使用。

## 4.3 浏览器目标

构建后应作为静态站点运行。核心功能不需要后端。

持久化优先本地：

- 保存已选择批注。
- 保存数值。
- 保存已访问节点。
- 保存已解锁结局。
- 提供重置进度按钮。

# 5. 领域模型

```ts
export type GameStats = {
  officialness: number;      // 正本度：官方路线 / 机构接受度
  marginalia: number;        // 旁注度：非官方声音密度
  residualVoice: number;     // 余声：无名者被保留的痕迹
```

# 6. 游戏系统

## 6.1 主地图

地图应表现为一张长卷。

必需功能：

- 从长安通向灵山的横向路线。
- 节点按文学逻辑放置，而非真实地理坐标。
- 图层切换：
  - 小说路线
  - 历史幽灵路线
  - 非官方旁注路线
  - 天庭 / 灵山调度层
  - 地下 / 妖洞 / 水府层
  - 梦境层
- 已访问节点需要有明显标记。
- 有未处理审稿事件的节点应出现印章或旗标。
- 余声很强的节点应长出边缘支线。
- 驳杂度很高的节点应在视觉上分裂出多条路线痕迹。

## 6.2 节点档案

玩家点击节点后，显示档案面板。

必需部分：

1. **正本**：章节范围、官方回目、摘要。
2. **路线**：在旅程中的位置、前后节点。
3. **门槛**：门槛类型与解释框架。
4. **非官方问题**：一个尖锐的文学 / 伦理问题。
5. **旁证**：证据物件。
6. **声音**：本节点可听到的声音。
7. **批注**：批注卡。
8. **审稿**：当前或潜在审稿意见。
9. **小游戏 / 交互**：若有。
10. **资料线索**：资料来源引用与说明。

## 6.3 批注放置

玩家应能放置或应用批注。

实现方式：

- 若稳定，可实现完整拖拽。
- 若拖拽脆弱，使用“点击选择 + 点击放置区域”。

放置区域：

- `mainText`：将批注吸收进官方正文。
- `margin`：作为可见边注保存。
- `hiddenArchive`：藏入档案，避开审稿，但之后可能被发现。
- `reviewPile`：提交给官方审稿。
- `blankSpace`：不解释，只保留沉默。

每种放置会以不同方式改变数值。

示例逻辑：

- 将 `corpse-note` 放入边注：`marginalia +2`，`residualVoice +2`，`officialness -1`，`reviewRisk +1`。
- 将同一张批注放入正文：`officialness +1`，`residualVoice +1`，`contradiction -1`。
- 将其放入留白处：`blankSense +2`，`residualVoice +1`，`routeClarity -1`。

## 6.4 审稿系统

审稿事件是本项目的核心讽刺引擎。

它们的语气应当官僚、礼貌、道德上回避。

审稿意见示例：

- “建议删除尸体细节。此处重点应为唐僧得救与取经意志坚定。”
- “白龙马第一人称自述过强，可能影响交通工具属性识别。”
- “陈家庄儿童祭祀部分不宜扩大为制度性问题，可归入地方风俗。”
- “狮驼岭伤亡数字过大，建议改为‘妖氛甚重’。”
- “女儿国不宜作为独立政治实体展开，建议保留为情关节点。”
- “无字经解释过多将损害不可言说性，解释过少将影响课程项目验收。”

玩家选择：

- 接受修改。
- 拒绝修改。
- 隐藏批注。
- 转为隐喻。
- 转为留白。
- 附加反证。

## 6.5 数值展示

数值展示应像文书系统，而不是普通 RPG HUD。

建议标签：

- 正本度
- 旁注度
- 余声
- 驳杂
- 审稿风险
- 留白感
- 慈悲
- 路线清晰度

使用简短 tooltip 解释每个数值。

## 6.6 结局

根据数值、已访问节点和批注选择实现结局解析器。

必需结局：

1. **正经图**：地图干净、被接受，但精神上空洞。
2. **万注本**：地图拥挤到不可阅读，但许多人终于在其中找到自己。
3. **无字图**：玩家理解何时应留下空白。
4. **浪浪山档案馆**：小妖没有抵达灵山，却建起路边档案馆。
5. **审稿失败**：地图因过于驳杂被退回，而退稿成为另一种流传。
6. **功德账本**：玩家优化了官方功德，却发现功德核算取代了慈悲。
7. **幽灵路线**：历史路线与小说路线无法重合，生成幽灵地理。
8. **删余声**：所有边缘声音都被删除，终页清晰、权威而闹鬼。

结局必须包含文学性正文，不能只有数值摘要。

# 7. 内容圣经

## 7.1 中心问题

内容围绕这些问题构建：

1. 谁有资格成为路线的一部分？
2. 当怪物变成隐喻，真实受害者会发生什么？
3. 地图是认识工具，还是管理工具？
4. 神圣旅程是否需要附带损耗？
5. 空白是缺席、审查、欺骗、超越，还是伦理克制？
6. 小妖能否拒绝模仿英雄，转而成为批注者？
7. 官方真相是虚假的，还是仅仅不完整？
8. 给某人名字是在保存他，还是覆盖其不可知性？

## 7.2 三种文本声部

使用三种主要散文风格。

### A. 官方文体

章回体 + 官僚式简洁。

示例：

> 第十三站，双叉岭。唐僧出界，误入虎穴，金星解厄，伯钦留僧。本站取经进度：一。随行损耗：不计。

### B. 非官方文体

地方性的、身体性的、碎片化的，有时带口述感。

示例：

> 我们只看见那个和尚活着出来。另外两个呢？老人说别问，问了山里会起风。可他们的鞋还在。

### C. 批注员文体

冷静、精确、略有伤痕感，观察尖锐。

示例：

> 地图在此处要求一条直线。但死亡总是旁逸斜出。我试着把两个无名者标上去，系统提示：比例尺不支持。

## 7.3 反复意象

使用反复意象统一作品：

- 留在路上的鞋。
- 空白名牌。
- 不肯保持笔直的路线。
- 像伤口一样的朱印。
- 连接证据卡的线绳。
- 反复出现的短句：“未统计。”
- 反复出现的短句：“不宜展开。”
- 反复出现的短句：“二名不是名字。”
- 反复出现的问题：“路过去以后，山还热不热？”
- 反复出现的终局图像：“纸的呼吸。”

## 7.4 小妖批注队

游戏可以包含四名小妖批注员。他们受“被忽略的小存在试图进入或改写宏大叙事”这一广义思路启发。

不要把他们只写成笑料。他们是项目的情感视角。

建议队伍：

### 小猪妖档案员

- 想被纳入某种宏大叙事。
- 整理官方回目，但私人情绪总会漏进记录。
- 机制功能：提升 `routeClarity`，也可能解锁官方摘要内部隐藏的悲伤。

### 蛤蟆精测绘员

- 技术型、紧张，总是画歪地图。
- 机制功能：通过“错误”地图 overlay 揭示隐藏支线和偶然真相。

### 黄鼠狼传闻员

- 话多、灵活、不可靠，但极懂社会关系。
- 机制功能：解锁谣言和误注，提高 `contradiction`。

### 猿猴临摹员

- 崇拜孙悟空，却害怕自己只会成为模仿品。
- 机制功能：识别英雄阴影和身份矛盾。

核心观念：

> 小妖不必成为英雄。小妖可以写下一条脚注，从而改变英雄被阅读的方式。

# 9. 小游戏 / 互动模块规格

实现若干嵌入式场景。它们应小而有意义。

## 9.1 遗物整理：双叉岭

场景：一个破损行囊。

物件：

- 破鞋。
- 半块干粮。
- 未寄出的家书。
- 血迹路线草图。
- 无名通关文牒碎片。
- 骨片。
- 空白名牌。

玩家选择：

- 归档为“随从二名”。
- 发明名字。
- 保存为未知者。
- 隐藏遗骸。
- 留下空白纪念牌。

数值效果：

- 命名可提高 `residualVoice`，也可能提高 `contradiction`。
- 归档为“随从二名”提高 `routeClarity` 和 `officialness`，但降低 `residualVoice`。
- 空白纪念提高 `blankSense`，并保留伦理张力。

## 9.2 紧箍合约：五行山

场景：“取经工程”与孙悟空之间可编辑的合约。

条款：

- 行动自由；
- 对使命的服从；
- 纠偏咒语；
- 暴力责任；
- 质疑命令的权利；
- 拒绝象征性暴力的权利。

玩家修改会影响后续数值：

- 严格合约：高路线清晰度，高怨气。
- 人道合约：更多驳杂，更多慈悲。

## 9.3 图标选择：鹰愁涧

选择白龙马在地图上的图标：

- 龙。
- 马。
- 罪人。
- 交通工具。
- 被赦免者。
- 沉默的第五位取经者。

选择会改变后续 UI 标签，以及白龙马声音碎片是否出现。

## 9.4 袈裟账簿：观音院

调查一件圣物如何穿过欲望、盗窃、赞赏、所有权和制度权威。

UI 设想：

- 一张账簿表格，列为：持有者、动机、公开理由、隐藏欲望、后果。

## 9.5 证词拼接：高老庄

来自以下对象的矛盾陈述：

- 高家。
- 村民。
- 猪八戒。
- 雇工。
- 小妖传闻员。

玩家拼接一份叙述。没有任何单一叙述完全真实。

## 9.6 名字陷阱：平顶山

屏幕上出现名字：

- 孙行者
- 弼马温
- 齐天大圣
- 猴子
- 斗战者
- 你

点击或承认某个名字可能触发捕获。交互应让玩家理解：命名是一种权力。

## 9.7 入境表：女儿国

一张官僚边境表格要求玩家分类身体、性别、法律身份、使命、欲望和外交假设。

如果玩家按外来者默认假设填写，表格会显示女儿国官员的反驳意见。

## 9.8 火焰山风向谜题

网格地图：

- 火焰格。
- 村庄格。
- 路线格。
- 隐藏档案格。
- 芭蕉扇方向控制。

风向效果：

- 西风打开路线。
- 东风损害村庄。
- 上风揭开天庭事故记录。
- 下风揭开家庭怨恨与地方记忆。

胜利条件同时考虑路线打开和附带损害。

## 9.9 狮驼岭亡者名册

一张亡者表不断被系统自动压缩。

系统转换：

- “姓名不详 37” → “无名者若干”
- “村落三处” → “地方损耗”
- “无法收殓” → “妖氛甚重”

玩家可选择抵抗、接受、隐藏，或转为留白纪念。

## 9.10 灵山无字经校验

玩家需要解释空白页。

可能含义：

- 欺骗。
- 审查。
- 超越。
- 删除。
- 等待未来书写。
- 拒绝过度解释的伦理。

最终结果取决于玩家此前如何使用留白注。

# 10. 美术方向

## 10.1 主视觉风格

古籍 / 木刻 / 档案拼贴。

基础色板：

- 纸张：暖米色 / 旧象牙色。
- 墨色：深棕 / 黑。
- 印章：朱砂红。
- 审稿标记：官僚蓝黑与红色删除线。
- 隐藏档案：灰、灰烬、褪色墨。
- 梦境层：稀释的蓝紫色或模糊水墨。

使用 CSS variables：

```css
:root {
  --paper: #f4ecd8;
  --paper-deep: #e2d2ad;
  --ink: #241a12;
  --ink-soft: #59483a;
  --seal: #a63225;
  --review-blue: #334b68;
  --ash: #77706a;
  --thread: #8c2f24;
}
```

不要依赖自定义字体文件。使用系统中文宋体 / 黑体字体栈。

建议字体栈：

```css
--font-serif: "Noto Serif SC", "Songti SC", "SimSun", serif;
--font-sans: system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
--font-mono: ui-monospace, SFMono-Regular, Menlo, Consolas, monospace;
```

## 10.2 UI 意象

- 被接受的官方记录用红印。
- 非官方路线支线用歪斜铅笔线。
- 印章文字可包括“不宜展开”“未统计”“材料缺失”“另存边档”。
- 批注卡使用撕裂纸片视觉。
- 用线绳 overlay 连接证据关系。
- 高风险批注背后出现墨迹渗开。
- 未知姓名使用空白矩形牌位。

## 10.3 动画规则

使用克制动画：

- 纸张展开。
- 印章落下。
- 路线线条绘制。
- 墨迹扩散。
- 边注滑入。
- 审稿意见盖在文字上。

避免过度游戏化粒子特效。动画应像纸、墨、印、风和档案翻动。

## 10.4 响应式设计

桌面：

- 地图在左 / 中。
- 档案在右。
- 数值在顶部或底部文书条。

移动端：

- 地图横向滚动。
- 档案作为底部抽屉打开。
- 批注放置使用点击菜单，而不是拖拽。

## 10.5 最终美观打磨要求

完成核心内容、核心玩法、测试和构建之后，必须进行一轮专门的视觉与美观打磨。该阶段不是新增大功能，而是提升作品的完整度和展示效果。

至少检查并优化：

- 首页第一屏是否具有课程展示级吸引力，能快速传达“会争吵的取经地图”这一概念。
- 主地图是否有卷轴感、层次感和纸张质感，节点、路线、支线、印章和裂痕是否有统一风格。
- 档案面板是否像真实文书 / 档案袋，而不是普通卡片列表。
- 批注卡是否有不同类型的视觉区分，同时保持中文可读性。
- 审稿意见是否有蓝黑批改、红色删线、盖章、退回意见等官僚视觉气质。
- 空白名牌、线绳、墨迹、纸角、折痕、污渍等细节是否适度出现，强化文学氛围。
- hover、focus、选中、已访问、已审稿、已归档等状态是否清楚且美观。
- 桌面端与移动端是否都保持可读、可演示、不卡顿。
- 不要为了美观牺牲可访问性、性能和文本阅读体验。

完成视觉打磨后，更新 `docs/ROADMAP.md` 或 README，简要记录已完成的美术优化与仍可继续提升的美术项。

# 11. UX 与可访问性

- 所有交互控件必须能通过键盘访问。
- 每个节点必须有 accessible label。
- 不要只靠颜色表达含义。
- 批注类型必须有文字标签和图标。
- 使用 `prefers-reduced-motion` 支持减少动画。
- 文本必须可读；不要为了古风牺牲可读性。
- 提供带确认的重置进度按钮。
- 提供“显示所有已解锁文本”的档案视图。
- 提供用于课程演示的“presentation mode”。

# 13. 资料与研究处理

## 13.1 资料引用

每个节点应有 `sourceRefs`，指向 `sources.ts` 中的条目。

资料条目字段：

```ts
export type SourceRef = {
  id: string;
  title: string;
  type: 'primary' | 'historical' | 'scholarly' | 'adaptation' | 'design-reference';
  note: string;
  url?: string;
};
```

## 13.2 应建模的主要资料

为以下内容建立 source refs：

- 《西游记》章节范围与事件。
- 《大唐西域记》作为历史地理对照。
- 有用时加入早期 / 相关西游传统。
- 当代改编作为设计影响，尤其是小人物重述与黑暗神话档案美学。

如果暂时无法获得精确引文，不要阻塞实现。使用资料摘要，并保留后续补充引用空间。

## 13.3 引文规则

不要用长篇复制文本填充项目。使用短摘录、转述和章节引用。创意价值应来自交互、批注和解释。

# 16. 内容写作示例

使用以下质量标准撰写游戏内文本。

## 16.1 双叉岭官方文本

> 第十三站，双叉岭。唐僧出界，误入虎穴，金星解厄，伯钦留僧。本站取经进度：一。随行损耗：未统计。

## 16.2 双叉岭边注文本

> 这里是取经路真正开始的地方。不是因为唐僧遇见了妖怪，而是因为地图第一次学会省略尸体。官方路线写：唐僧遇难。山风写：三个人上路，一个人被救。二名不是名字。随从也不是人生。

## 16.3 火焰山边注文本

> 火不是从山里开始的。火从一次天庭事故、一段旧怨、一件法宝、一场婚姻破裂和一群被热死的普通人之间开始。官方地图只画一座红山。村民地图画井、坟、迁徙线和卖水的人。路要过去。可路过去以后，山还热不热？

## 16.4 狮驼岭边注文本

> 系统建议将三万七千个名字合并为“妖氛甚重”。我点击撤销。系统提示：撤销失败，地图空间不足。

## 16.5 灵山留白文本

> 你没有替他们说完。你只在他们消失的地方，留下纸的呼吸。

# 17. 数据质量规则

每个路线节点应有：

- 唯一 `id`。
- 清晰章节范围。
- 尖锐非官方问题。
- 至少一段文学注解。
- 至少一个声音。
- 至少一个资料来源引用。
- 重要节点有多张批注卡。
- 伦理张力强的节点至少有一个审稿风险或审稿事件。

每张批注卡应有：

- 类型。
- 发言者。
- 有力文本。
- 可放置区域。
- 数值效果。
- 标签。

每个结局应有：

- 明确解锁逻辑。
- 文学性正文。
- 地图视觉转化。
- 与项目中心前提的关系。

# 18. 实现优先级

从零构建时按此顺序优先：

1. 项目初始化与可运行构建。
2. 领域类型与数据模型。
3. 路线节点和批注的丰富种子数据。
4. 主地图 UI。
5. 节点档案 UI。
6. 批注放置与数值引擎。
7. 审稿引擎。
8. 持久化。
9. 结局解析器与结局页面。
10. 档案馆视图。
11. 小游戏 / 互动模块。
12. 视觉打磨。
13. 测试。
14. 文档。
15. 进一步内容扩充。
16. 在主体功能、测试和文档稳定后，进行专门的美观提升：统一视觉系统、补足微交互、优化布局、增强地图与档案质感，使作品更适合课程展示。

若必须取舍，保留概念核心：路线地图 + 批注 + 审稿 + 结局 + 丰富文本。

# 19. 避免这些错误

- 不要把项目做成战斗 RPG。
- 不要让玩家只是“收集所有批注”而没有取舍。
- 不要把正本度自动写成邪恶，也不要把旁注度自动写成善。好的设计必须有张力。
- 不要把小妖简化成笑料。
- 不要让地图变成扁平章节列表。
- 严肃节点不要过度使用现代网络口语。
- 不要让所有批注都像同一个声音。
- 不要使用巨型图片或自定义字体文件，导致项目难以运行。
- 游戏本体是中文时，不要让 UI 标签留在英文，除非它们是内部开发标识。
- 不要创建玩家无法理解的隐藏机制。
- 不要在完成主体后忽视美术与美观；作品应在视觉上有统一、克制、可展示的完成度。
